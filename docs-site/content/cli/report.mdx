# packguard report

Evaluates every installed dependency against `.packguard.yml` and prints a compliance table. Reads **only** the local store вҖ” zero network calls.

## Synopsis

```bash
packguard report [path] \
  [--format table|json|sarif] \
  [--fail-on-violation] \
  [--show-policy] \
  [--project <workspace>]
```

## Output shape

Compliance table grouped by ecosystem вҶ’ workspace вҶ’ package:

| Column | Values |
|---|---|
| `Policy` | `compliant` В· `warning` В· `violation` В· `cve-violation` В· `malware` В· `typosquat` В· `insufficient` |
| `Risk` | Combined badges: `2рҹ”ҙ В· 1рҹҹ  В· 1рҹҸҙвҖҚвҳ пёҸ В· вҡ ` вҖ” CVE counts + malware confirmed + typosquat suspects |

**Footer** вҖ” summary line:

```
compliant 19 В· warnings 80 В· violations 9 В· insufficient 10
vulnerabilities: 4 critical В· 4 high В· 0 medium В· 0 low
supply-chain: рҹҸҙвҖҚвҳ пёҸ 0 malware confirmed В· вҡ  1 typosquat suspect
```

## Flags

| Flag | Effect |
|---|---|
| `--format table\|json\|sarif` | Default `table`. |
| `--fail-on-violation` | Exit 2 when any row is `violation`, `cve-violation`, or `malware`. |
| `--show-policy` | Print the resolved policy with per-key provenance (which `.packguard.yml` and line set each value) and exit. See [Policy cascade](/concepts/offset-policy#policy-cascade-across-a-monorepo). |
| `--project <path>` | Scope to a single workspace in a monorepo. |

Past the table, the command prints a pedagogical footer bridging to `packguard audit` вҖ” and when packages surface as `insufficient`, it points you at `--show-policy` so you can see which cascade layer is constraining resolution.

## Examples

```bash
# Human-readable, all scopes.
packguard report

# CI gate вҖ” fail on any blocking policy hit.
packguard report --fail-on-violation

# SARIF for inline annotations in the Security tab.
packguard report --format sarif > packguard.sarif

# Scope to a single workspace.
packguard report --project ./apps/web
```

## `report` vs `audit`

Two commands, two questions:

- **`audit`** вҖ” *"what are the known risks?"* Lists every CVE, malware, typosquat hit.
- **`report`** вҖ” *"how do those risks interact with our policy?"* Evaluates every row against `.packguard.yml` and flags the ones that violate the configured block rules.

A package with a `high`-severity CVE shows up in both, but `audit` says "this CVE exists" and `report` says "this CVE is blocking under your current `block.cve_severity`". For CI gating, `report --fail-on-violation` is usually what you want вҖ” it lets you tune the bar by editing YAML instead of toggling CLI flags.
